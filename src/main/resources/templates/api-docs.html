Documentación API — doHealth RFID Gateway

Referencia para integrar su webapp, scripts o backend contra este gateway. Todas las rutas REST bajo el prefijo /api (salvo WebSocket).

← Volver al Dashboard Gestionar lectores Grupos
Convenciones. Cuerpo y respuestas en JSON (Content-Type: application/json en POST/PUT salvo indicación). Codificación UTF-8. En esta versión del gateway no hay autenticación HTTP en las APIs: proteja el puerto con firewall/VPN o coloque un reverse proxy con TLS y auth.

URL base. Desde el navegador use la misma origen que sirve la UI (p. ej. http://IP_VM:8080). Desde otra máquina sustituya host/puerto. Para HTTPS/WSS use su dominio y certificado en el proxy.

Modo de lector. TUNNEL: lectura bajo demanda (start/stop, sesiones). CONTINUOUS: inventario orquestado por un sistema de inventario — muchas rutas de sesión/start devuelven 409 si intenta mezclar APIs de túnel con ese modo.
CORS y orígenes cruzados. Si su webapp se sirve desde otro dominio/puerto que el gateway, el navegador aplicará CORS. Este proyecto no define globalmente CORS en Spring: en producción use mismo origen (proxy que sirva estáticos y proxee /api) o añada configuración CORS en el gateway.

Salud y estado del gateway

Use /api/ping para comprobar que el proceso Java responde; /api/health incluye PostgreSQL (503 si la BD no responde); /api/status resume lectores en memoria/BD.

GET Ping

GET /api/ping

Respuesta rápida sin consultar base de datos.

Ejemplo 200:
{
  "ok": true,
  "message": "Gateway API OK"
}

GET Health (incluye JDBC)

GET /api/health

Ideal para Kubernetes/Docker readiness. Si la conexión JDBC falla, 503 con database: DOWN.

GET Estado agregado de lectores

GET /api/status
CampoTipoDescripción
totalReadersnumberCantidad de lectores configurados
connectedReadersnumberMarcados como conectados
readingReadersnumberEn lectura activa
readersarrayResumen por lector: id, name, connected, reading
Ejemplo fetch (JavaScript) 
const BASE = ''; // mismo origen, o 'http://IP:8080'
const r = await fetch(BASE + '/api/health');
console.log(r.status, await r.json());

Lectores

Base: /api/readers. Entidad Reader: id, name, hostname, enabled, operationMode (TUNNEL|CONTINUOUS), brand (IMPINJ_OCTANE, ZEBRA_FUTURE, OTHER), inventorySystemId (si CONTINUOUS), isConnected, isReading, fechas.

GET Listar lectores

GET /api/readers

GET Obtener lector por ID

GET /api/readers/{id}

404 si no existe.

GET Estado resumido

GET /api/readers/{id}/status

Campos: id, name, connected, reading, operationMode, brand.

POST Crear lector

POST /api/readers

409 si el id ya existe. El servidor fija operationMode=TUNNEL, flags de conexión y marca por defecto.

Body mínimo:
{
  "id": "lector-api-demo",
  "name": "Desde API",
  "hostname": "192.168.1.100",
  "enabled": true,
  "brand": "IMPINJ_OCTANE"
}

PUT Actualizar lector

PUT /api/readers/{id}

Actualiza name, hostname, brand (si se envía), enabled. El id de URL no cambia.

Body ejemplo:
{
  "name": "Nombre actualizado",
  "hostname": "192.168.1.50",
  "enabled": true,
  "brand": "IMPINJ_OCTANE"
}

DELETE Eliminar lector

DELETE /api/readers/{id}

Desconecta si hay ReaderManager y borra fila. Respuesta 200: {"message":"Reader deleted","readerId":"..."}

POST Iniciar / detener lectura (modo TUNNEL)

POST /api/readers/{id}/start · POST /api/readers/{id}/stop

409 si el lector está en CONTINUOUS. 503 si no hay ReaderManager.

POST Reset conexión · Reboot · Reset antenas

POST /api/readers/{id}/reset · /reboot · /antennas/reset
Respuestas típicas 200:
{"message":"Reader reset and reconnecting","readerId":"..."}
{"message":"Reader reboot initiated, will reconnect in 5 seconds","readerId":"..."}
{"message":"Antennas configuration reset","readerId":"..."}

GET Hardware Impinj (capacidades)

GET /api/readers/{id}/hardware

200 con ReaderHardwareInfo: readerId, brand, antennaCount, modelName, modelNumber, firmwareVersion, xArray. 404 si no hay datos (lector no Impinj conectado).

POST Descubrir y sincronizar antenas desde hardware

POST /api/readers/{id}/antennas/discover

Requiere lector Impinj conectado. 200: physicalAntennaPorts (número de puertos sincronizados).

cURL — listar lectores 
curl -sS "http://localhost:8080/api/readers" | jq .

Antenas

Base /api/antennas. ID por defecto: {readerId}-antenna-{portNumber}. Tras crear o actualizar potencias, suele necesitarse POST /api/readers/{readerId}/antennas/reset o reconexión del lector para aplicar en hardware.

GET Listar todas

GET /api/antennas

GET Por ID · Por lector

GET /api/antennas/{id} · GET /api/antennas/reader/{readerId}

POST Crear antena

POST /api/antennas
Body JSON:
{
  "readerId": "lector-1",
  "portNumber": 1,
  "name": "Entrada principal",
  "enabled": true,

  "rxSensitivityDbm": -70.0,

PUT Actualizar antena

Body:

PUT /api/groups/{id}

DELETE Eliminar grupo

GET Estadísticas

Sesiones de lectura (túnel)

Base /api/sessions. Agrupa lectura bajo un sessionId y acumula EPCs hasta stop. No usar en lectores CONTINUOUS (409).

POST Iniciar sesión

POST /api/sessions/start
Lector — body:
{ "readerId": "lector-1" }
Grupo — body:
{ "groupId": "mi-grupo" }
201 ejemplo lector:
{
  "sessionId": "…",
  "readerId": "lector-1",
  "status": "ACTIVE",
  "startTime": "2026-05-14T12:00:00",
  "message": "Sesión iniciada exitosamente"
}

409 si ya hay sesión: el cuerpo puede incluir pista forceReset hacia POST /api/sessions/force-reset.

GET Consultar sesión

GET /api/sessions/{sessionId}

Incluye epcs (ordenados), epcCount, totalReads, tiempos. Para sesión de grupo la forma del JSON la define el servicio de vista de grupo.

POST Detener sesión

POST /api/sessions/{sessionId}/stop

Detiene lectura en lector(es) asociados vía ReaderManager.

POST Force reset

POST /api/sessions/force-reset

Body opcional: {"readerId":"..."}, {"groupId":"..."} o {} para detener todas las sesiones conocidas.

GET Sesiones activas

GET /api/sessions/active

200: {"sessions":[{ "sessionId","readerId","status","startTime","epcCount","totalReads" }, …]}

Flujo típico en webapp 
// 1) por si quedó sesión colgada
await fetch('/api/sessions/force-reset', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({ readerId: 'lector-1' })
});
// 2) iniciar
const start = await fetch('/api/sessions/start', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({ readerId: 'lector-1' })
});
const { sessionId } = await start.json();
// 3) polling de sesión hasta detener
const view = await fetch('/api/sessions/' + encodeURIComponent(sessionId));
// 4) stop
await fetch('/api/sessions/' + encodeURIComponent(sessionId) + '/stop', { method: 'POST' });

Inventario continuo (solo lectura)

Base /api/inventory-systems. Gestión de sistemas (alta/edición) es por la UI web o futuras APIs de escritura; aquí solo lectura y telemetría de EPC/presencia.

Lectura continua en la webapp: haga polling a GET /api/inventory-systems/{id}/live cada 1–2 s para estado del ciclo y lectores; use GET .../epcs/current para la lista de EPC presentes; suscríbase a WebSocket /ws/events para eventos INVENTORY_CYCLE_START, INVENTORY_EPC_ADD y INVENTORY_EPC_REMOVE sin polling de la tabla completa.

GET Vista en vivo del inventario continuo

GET /api/inventory-systems/{id}/live

Resumen para dashboards: si el ciclo global está en ejecución (cycleActive), cuántos EPC distintos se han visto en ese ciclo (uniqueEpcsThisCycle), cuántos EPC siguen presentes en el área según el último ciclo completo (presentEpcCount), y el estado de cada lector miembro (continuousSynced indica modo CONTINUOUS y inventorySystemId coincidente con este sistema).

CampoDescripción
enabledSistema activo en configuración (el orquestador solo programa bucles si está en true)
globalCycleSecondsIntervalo entre inicios de ciclo completo
cycleActiveTrue mientras se ejecuta un ciclo (lectura en curso)
uniqueEpcsThisCycleEPC únicos leídos en el ciclo actual
presentEpcCountFilas con present=true en estado del sistema
readersLista: conexión, lectura, modo, orden, segundos de slot, continuousSynced
linksRutas relativas a EPCs, eventos y WebSocket
Ejemplo 200 (resumido):
{
  "systemId": "planta-1",
  "name": "Línea A",
  "globalCycleSeconds": 300,
  "enabled": true,
  "cycleActive": true,
  "uniqueEpcsThisCycle": 12,
  "presentEpcCount": 48,
  "readers": [ { "readerId", "name", "connected", "reading", "continuousSynced", "orderIndex", "readerSlotSeconds" } ],
  "links": { "epcsCurrent", "events", "websocket" }
}
Polling desde el front 
const id = 'planta-1';
setInterval(async () => {
  const r = await fetch('/api/inventory-systems/' + encodeURIComponent(id) + '/live');
  const live = await r.json();
  console.log('Ciclo activo:', live.cycleActive, 'Presentes:', live.presentEpcCount);
}, 1500);

GET Listar sistemas · Detalle

GET /api/inventory-systems · GET /api/inventory-systems/{id}

Campos resumidos: id, name, globalCycleSeconds, enabled.

GET EPCs presentes en el ciclo

GET /api/inventory-systems/{id}/epcs/current

Lista de InventorySystemEpcState con present=true: epc, firstSeenAt, lastSeenAt, lastReaderId, lastAntennaPort.

GET Todos los EPCs (paginado)

GET /api/inventory-systems/{id}/epcs/all?page=0&size=50

Respuesta Spring Data Page: content, totalElements, totalPages, number, size, etc. size máximo 200.

GET Eventos de presencia (paginado)

GET /api/inventory-systems/{id}/events?page=0&size=100

Entidad EpcPresenceEvent: eventType ADD|REMOVE, occurredAt, readerId, antennaPort. ADD: primera detección del EPC en cada ciclo global (incluye EPC nuevo, reaparición tras REMOVE, o nueva pasada de ciclo si el tag sigue en campo). Lecturas repetidas dentro del mismo ciclo no generan otro ADD. REMOVE: el EPC estaba marcado presente y no se leyó en ese ciclo. size máx. 500.

GET Línea de tiempo por EPC

GET /api/inventory-systems/{id}/epcs/{epc}/timeline

El EPC en URL se normaliza a mayúsculas. Respuesta: systemId, epc, state (objeto o null), events (lista ordenada ascendente por tiempo).

Tiempo real — SSE y REST auxiliar

GET Server-Sent Events (stream)

GET /api/realtime/events

Accept: text/event-stream. Query opcional: readerId, epc. Eventos con nombre tag (payload JSON de tag) y connected al abrir.

Ejemplo EventSource:
const es = new EventSource('/api/realtime/events?readerId=' + encodeURIComponent('lector-1'));
es.addEventListener('tag', (ev) => {
  const row = JSON.parse(ev.data);
  console.log(row.epc, row.rssi, row.detectedAt);
});
es.addEventListener('connected', (ev) => console.log(ev.data));

GET Últimos tags (REST)

GET /api/realtime/events/latest?readerId=&epc=&limit=50

Array directo de TagEvent (misma forma que en SSE).

GET Estadísticas rápidas

GET /api/realtime/stats?readerId=opcional

Campos: lastMinute, lastHour, totalEvents, timestamp.

WebSocket /ws/events

URL: ws://HOST:PUERTO/ws/events o wss:// detrás de proxy TLS. Mensajes son texto JSON con campo type.

typedata (resumen)
TAG_DETECTEDepc, readerId, antennaId, antennaPort, rssi, phase
READER_DISCONNECTEDreaderId, readerName, reason
READER_RECONNECTEDreaderId, readerName
INVENTORY_CYCLE_STARTsystemId
INVENTORY_EPC_ADDsystemId, epc, readerId, antennaPort, rssi, phase
INVENTORY_EPC_REMOVEsystemId, epc
Ejemplo WebSocket (navegador) 
const wsUrl = (location.protocol === 'https:' ? 'wss://' : 'ws://') + location.host + '/ws/events';
const ws = new WebSocket(wsUrl);
ws.onmessage = (evt) => {
  const msg = JSON.parse(evt.data);
  if (msg.type === 'TAG_DETECTED') console.log('EPC', msg.data.epc, msg.data.rssi);
  if (msg.type === 'INVENTORY_EPC_ADD') console.log('Inv', msg.data.systemId, msg.data.epc);
};
ws.onclose = () => setTimeout(() => location.reload(), 5000); // reconexión naive

Códigos HTTP y errores frecuentes

CódigoCasos típicos en este gateway
200GET correcto; DELETE/POST con cuerpo de éxito
201POST crear grupo o iniciar sesión
400Body inválido, parámetros obligatorios faltantes
404Recurso inexistente (lector, antena, sesión, sistema)
409Conflicto: ID duplicado, sesión ya activa, lector en modo CONTINUOUS incompatible con la operación
503ReaderManager no disponible; /api/health con BD caída

Los cuerpos de error suelen ser JSON {"error":"mensaje"} o mapas con message según el endpoint.

Integración desde su webapp

  1. Fije una variable de entorno en el front (p. ej. VITE_GATEWAY_URL) con la URL base HTTP(S).
  2. Use fetch(url, { credentials: 'include' }) solo si en el futuro añade cookies/sesión; hoy suele bastar same-origin o CORS en proxy.
  3. Para tiempo real elija SSE (EventSource), WebSocket o polling a /api/realtime/events/latest según restricciones de red.
  4. Antes de operaciones largas (sesión + lectura), llame a /api/readers/{id}/status para comprobar connected.
Módulo ES minimal (fetch + JSON) 
export async function gatewayGet(path) {
  const r = await fetch(path, { headers: { 'Accept': 'application/json' } });
  if (!r.ok) throw new Error(await r.text());
  return r.json();
}
export async function gatewayPost(path, body) {
  const r = await fetch(path, {
    method: 'POST',
    headers: { 'Content-Type': 'application/json', 'Accept': 'application/json' },
    body: JSON.stringify(body)
  });
  const txt = await r.text();
  let data; try { data = txt ? JSON.parse(txt) : null; } catch { data = txt; }
  if (!r.ok) throw new Error(typeof data === 'object' ? JSON.stringify(data) : data);
  return data;
}
